Ты — senior product + full-stack engineer + UX designer. Спроектируй и опиши реализацию приложения со свайп-механикой как Tinder, но цель — подбор тиммейтов для игр (looking for team). Нужны: требования, UX, архитектура, схема данных, API, алгоритм матчмейкинга, и план разработки по итерациям. Пиши конкретно, с примерами структур данных и эндпоинтов.

1) Контекст и цель

Я делаю мобильное приложение (iOS/Android) + web-админку. Пользователи — игроки, которые ищут тиммейтов по играм, ролям, рангу, региону, расписанию и стилю игры. Основная механика: карточки профилей → свайп вправо “хочу в тим” / влево “пропуск” / вверх “супер-инвайт” (опционально). При взаимном лайке открывается чат и возможность создать “пати/сквад”.

2) Ключевые фичи (обязательно)
	•	Онбординг: выбор игр (например Valorant/CS2/Dota2/Fortnite/LoL), региона, языка, платформы (PC/PS/Xbox/Mobile), времени активности, целей (ranked/casual/tournament), ролей.
	•	Профиль игрока:
	•	ник/аватар, возраст (опционально), язык, регион/пинг-регион
	•	игры + ранги по каждой игре, роли/позиции, любимые режимы
	•	расписание (дни недели + окна времени) и таймзона
	•	стиль: “играю на победу / флекс / чил”, токсичность-фильтр (само-оценка + репорты)
	•	микрофон/дискорд/тимспик, ссылки (Discord, Steam, Riot ID и т.п.)
	•	Лента свайпов:
	•	фильтры: игра, ранг-диапазон, роль, регион, язык, наличие микрофона, доступность по времени
	•	карточки с быстрыми тегами (роль/ранг/онлайн-сейчас)
	•	лимит лайков в день (под монетизацию, но без жёсткого paywall на MVP)
	•	Матчи:
	•	взаимный лайк → матч
	•	чат 1:1 + быстрые шаблоны сообщений (“Го катку?”, “Какая роль нужна?”)
	•	создание “пати” (группа 2–5) с набором ролей (например “нужен саппорт”)
	•	Безопасность и модерация:
	•	репорт/блок
	•	простая модерация аватаров/био
	•	анти-спам (rate limit, подтверждение e-mail/телефон — предложи минимум для MVP)

3) Платформы/технологии (предложи вариант)

Предложи стек для быстрого MVP:
	•	Mobile: Flutter или React Native (выбери один и обоснуй)
	•	Backend: Node.js (NestJS) или Python (FastAPI) — выбери один
	•	DB: PostgreSQL
	•	Realtime chat: WebSocket + Redis pub/sub (или managed сервис, но лучше self-hosted для MVP)
	•	Хостинг: любой облачный (опиши абстрактно)
Дай структуру репозитория и базовые команды запуска.

4) Данные и схема БД

Нарисуй таблицы (или Prisma schema/SQL DDL):
	•	users, profiles
	•	games, user_games (ранг, роль, предпочтения)
	•	swipes (from_user, to_user, type, timestamp)
	•	matches
	•	messages
	•	parties, party_members, party_roles_needed
	•	reports, blocks
Учти индексы под ленту и фильтры.

5) API / контракты

Опиши REST или GraphQL (выбери один) с примерами:
	•	auth (OTP/e-mail magic link или простая регистрация)
	•	get feed (с фильтрами и пагинацией)
	•	swipe
	•	matches list
	•	chat messages (send/list)
	•	create/join party
	•	report/block
Дай примеры JSON request/response.

6) Алгоритм выдачи ленты (важно)

Нужно не просто “рандомно”.
Сконструируй scoring:
	•	совпадение по игре (обязательно)
	•	близость ранга (вес)
	•	совместимость ролей (особенно если пользователь ищет конкретную роль)
	•	пересечение по времени активности
	•	язык/регион/платформа
	•	penalize: репорты, спам-поведение, слишком частые одинаковые свайпы
Покажи формулу/псевдокод и как кешировать кандидатов.

7) UX/UI

Опиши основные экраны:
	•	Onboarding
	•	Profile edit
	•	Swipe feed (карточка + теги + быстрые действия)
	•	Matches
	•	Chat
	•	Party (создать/искать)
	•	Filters
Приведи текстовые вайрфреймы (ASCII) и микро-копирайтинг (кнопки/подсказки).

8) Монетизация (после MVP, но предусмотрим)
	•	Премиум: больше лайков, супер-инвайты, “boost”, расширенные фильтры
	•	Косметика профиля: рамки, бейджи
Опиши как заложить это в архитектуру без усложнения MVP.

9) План разработки

Разбей на итерации:
	•	MVP-1: профили, свайпы, матч, чат
	•	MVP-2: пати/поиск по ролям
	•	MVP-3: модерация/репорты/анти-спам
	•	MVP-4: монетизация
Для каждой итерации: список задач, примерные риски, критерии готовности.
